# 02 — Authentication Guide

This document describes the complete authentication system: how tokens are issued, how server and client components read them, how refresh and logout work, and how to manage static users during development.

---

## 1. Overview

Authentication in this project is **session-token based**, not JWT-based. Tokens are opaque strings generated by `AuthService` and stored in a server-side `Map<string, User>`. There is no JWT signing, no expiry claim, and no secret key needed for validation. The trade-off: tokens are lost when the server restarts (acceptable in development; replace with a Redis-backed store in production).

The token is stored in two places on the client:

| Storage | Who reads it | How |
|---|---|---|
| Cookie (`mono_session`) | Next.js server components | `getServerSession()` reads the cookie |
| `sessionStorage` | Next.js client components | `useAuth()` reads `sessionStorage` |

Both are set at login and cleared at logout.

## 3. Login Flow

### What happens step by step

```
Client                              API Gateway (AuthService)
  │                                         │
  │──── POST /api/auth/login ──────────────►│
  │     { email, password }                  │
  │                                         │ findByEmail(email)
  │                                         │ bcrypt.compare(password, hash)
  │                                         │ issueToken(userId) → "sess_usr01_1234_abc"
  │                                         │ sessions.set(token, user)
  │◄─── 200 OK ─────────────────────────────│
  │     { user, token }                      │
  │                                         │
  │  Store: cookie + sessionStorage          │
```

### API call

```bash
curl -s -X POST http://localhost:3001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@mono.dev","password":"admin123"}' \
  | jq .
```

Response:

```json
{
  "success": true,
  "data": {
    "user": {
      "id": "usr_01",
      "email": "admin@mono.dev",
      "name": "Admin User",
      "roles": ["admin", "user"],
      "permissions": ["read", "write", "admin"]
    },
    "token": "sess_usr_01_1746532800000_k7x9p"
  },
  "message": "OK"
}
```

### Token format

```
sess_{userId}_{unixTimestampMs}_{9-char random alphanumeric}
```

Example: `sess_usr_01_1746532800000_k7x9p3z8m`

The format is deterministic in structure but the random suffix makes it non-guessable for the same user across restarts.

### Client-side storage (Next.js shell)

After login, the shell app stores the token in both places:

```typescript
// apps/shell/src/lib/auth/client.ts  (illustrative)

async function login(email: string, password: string) {
  const res = await fetch('/api/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ email, password }),
  });

  const body = await res.json();
  if (!body.success) throw new Error(body.message);

  const { user, token } = body.data;

  // 1. Store in sessionStorage — available to client components
  sessionStorage.setItem('mono_token', token);
  sessionStorage.setItem('mono_user', JSON.stringify(user));

  // 2. Store in cookie — available to server components
  // In practice this is set by the Next.js route handler, not client-side JS,
  // to keep the cookie HttpOnly.
  await fetch('/api/session', {
    method: 'POST',
    body: JSON.stringify({ token, user }),
  });
}
```

The cookie is set as `HttpOnly; SameSite=Lax` by the Next.js route handler so it cannot be read by client-side JavaScript directly — only the server reads it.

## 4. Reading the Token in Server Components — `getServerSession()`

Server components (React components that run only on the server in Next.js App Router) read the session from the cookie:

```typescript
// apps/shell/src/lib/auth/server.ts  (illustrative)
import { cookies } from 'next/headers';
import type { User } from '@mono/shared/types';

export interface ServerSession {
  user: User;
  token: string;
}

/**
 * Reads the current authenticated session from the HttpOnly cookie.
 * Returns null if the cookie is absent or malformed.
 *
 * Use in Server Components and Route Handlers only — not in client components.
 */
export async function getServerSession(): Promise<ServerSession | null> {
  const cookieStore = await cookies();
  const raw = cookieStore.get('mono_session')?.value;
  if (!raw) return null;

  try {
    return JSON.parse(raw) as ServerSession;
  } catch {
    return null;
  }
}
```

Usage in a server component:

```typescript
// apps/shell/src/app/dashboard/page.tsx
import { redirect } from 'next/navigation';
import { getServerSession } from '@/lib/auth/server';

export default async function DashboardPage() {
  const session = await getServerSession();
  if (!session) redirect('/login');

  const res = await fetch(`${process.env.API_URL}/api/payments`, {
    headers: { Authorization: `Bearer ${session.token}` },
    cache: 'no-store',
  });

  const body = await res.json();
  const payments = body.data.items;

  return <PaymentList payments={payments} />;
}
```

Key points:
- `getServerSession()` is safe to call in any server component — it uses Next.js's `cookies()` API which is server-only
- The token is passed as a `Bearer` header to the API Gateway for every protected request
- If `getServerSession()` returns null, redirect to `/login` — do not let unauthenticated users see protected pages

## 5. Reading the Token in Client Components — `useAuth()`

Client components (interactive React components with `'use client'`) read from `sessionStorage`:

```typescript
// apps/shell/src/lib/auth/use-auth.ts  (illustrative)
'use client';

import { useState, useEffect } from 'react';
import type { User } from '@mono/shared/types';

export interface AuthState {
  user: User | null;
  token: string | null;
  isLoaded: boolean;
}

/**
 * Reads the current session from sessionStorage.
 * Returns null values until the component has mounted (SSR-safe).
 *
 * Use in Client Components only — not in server components.
 */
export function useAuth(): AuthState {
  const [state, setState] = useState<AuthState>({
    user: null,
    token: null,
    isLoaded: false,
  });

  useEffect(() => {
    const token = sessionStorage.getItem('mono_token');
    const userRaw = sessionStorage.getItem('mono_user');
    const user = userRaw ? (JSON.parse(userRaw) as User) : null;
    setState({ user, token, isLoaded: true });
  }, []);

  return state;
}
```

Usage in a client component:

```typescript
// apps/shell/src/components/refund-button.tsx
'use client';

import { useAuth } from '@/lib/auth/use-auth';

export function RefundButton({ paymentId }: { paymentId: string }) {
  const { token, isLoaded } = useAuth();

  async function handleRefund() {
    if (!token) return;
    await fetch(`/api/payments/${paymentId}/refund`, {
      method: 'POST',
      headers: { Authorization: `Bearer ${token}` },
    });
  }

  if (!isLoaded) return null;
  return <button onClick={handleRefund}>Refund</button>;
}
```

Key points:
- `useAuth()` is SSR-safe — it only reads `sessionStorage` inside `useEffect`, after the component has mounted on the client
- Always check `isLoaded` before rendering anything that depends on auth state — on the server render, `user` and `token` will be `null`
- Use `useAuth()` only for UI decisions (show/hide buttons, display the user's name). For data fetching from server components, use `getServerSession()` instead

## 6. All Subsequent Requests

Every request to a protected API endpoint must include the `Authorization` header:

```
Authorization: Bearer sess_usr_01_1746532800000_k7x9p3z8m
```

`JwtAuthGuard` validates the token on every request:

```typescript
// Simplified from jwt-auth.guard.ts
const token = authHeader.replace('Bearer ', '');
const user = this.authService.validateToken(token);
// authService.validateToken does: return this.sessions.get(token) ?? null
if (!user) throw new UnauthorizedException('Invalid or expired token');
request.user = user;
```

The resolved `User` object is attached to `request.user`. Controllers access it via `@CurrentUser()`:

```typescript
@Get('me')
me(@CurrentUser() user: User): User {
  return user;
}
```

## 7. Token Refresh Flow

The refresh endpoint rotates the token: the old token is immediately invalidated and a new token is returned.

```
Client                              API Gateway
  │                                         │
  │──── POST /api/auth/refresh ────────────►│
  │     Authorization: Bearer <old-token>   │
  │                                         │ sessions.delete(oldToken)
  │                                         │ newToken = issueToken(user.id)
  │                                         │ sessions.set(newToken, user)
  │◄─── 200 OK ─────────────────────────────│
  │     { user, token: <new-token> }         │
  │                                         │
  │  Update: cookie + sessionStorage         │
```

### API call

```bash
curl -s -X POST http://localhost:3001/api/auth/refresh \
  -H "Authorization: Bearer $OLD_TOKEN" \
  | jq .
```

### Client-side handling

After receiving the new token, the shell must update both storage locations:

```typescript
async function refreshSession(oldToken: string) {
  const res = await fetch('/api/auth/refresh', {
    method: 'POST',
    headers: { Authorization: `Bearer ${oldToken}` },
  });

  const body = await res.json();
  if (!body.success) {
    // Session not found — redirect to login
    await logout();
    return;
  }

  const { user, token } = body.data;

  sessionStorage.setItem('mono_token', token);
  sessionStorage.setItem('mono_user', JSON.stringify(user));

  // Update HttpOnly cookie via route handler
  await fetch('/api/session', {
    method: 'POST',
    body: JSON.stringify({ token, user }),
  });
}
```

## 8. Logout Flow

Logout clears the server-side session, the cookie, and sessionStorage.

```
Client                              API Gateway
  │                                         │
  │──── POST /api/auth/logout ─────────────►│
  │     Authorization: Bearer <token>       │
  │                                         │ sessions.delete(token)
  │◄─── 200 OK { ok: true } ───────────────│
  │                                         │
  │  Clear: cookie + sessionStorage          │
  │  Redirect to /login                      │
```

### API call

```bash
curl -s -X POST http://localhost:3001/api/auth/logout \
  -H "Authorization: Bearer $TOKEN" \
  | jq .
```

Response:

```json
{
  "success": true,
  "data": { "ok": true },
  "message": "OK"
}
```

After logout, the old token is invalid. Any request using it returns 401.

### Client-side logout function

```typescript
async function logout() {
  const token = sessionStorage.getItem('mono_token');

  if (token) {
    // Tell the server to invalidate the session
    await fetch('/api/auth/logout', {
      method: 'POST',
      headers: { Authorization: `Bearer ${token}` },
    }).catch(() => {
      // Ignore network errors — still clear client state
    });
  }

  // Clear client storage regardless of server response
  sessionStorage.removeItem('mono_token');
  sessionStorage.removeItem('mono_user');

  // Clear the HttpOnly cookie via a Next.js route handler
  await fetch('/api/session', { method: 'DELETE' });

  // Redirect to login
  window.location.href = '/login';
}
```

The `catch()` on the server call is intentional: if the server is temporarily unreachable, the client still logs out locally by clearing storage.

## 9. How to Add a New Static User

Static users live in `apps/api-gateway/src/modules/auth/repositories/static-user.repository.ts`.

### Step 1 — Generate a bcrypt hash

```bash
# Using Node.js (bcryptjs is already installed):
node -e "const bcrypt = require('bcryptjs'); bcrypt.hash('mypassword', 10).then(h => console.log(h));"
```

Output: `$2a$10$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`

### Step 2 — Add the user to STATIC_USERS

```typescript
const STATIC_USERS: UserRow[] = [
  // ... existing users ...
  {
    id: 'usr_03',                                    // must be unique
    email: 'developer@mono.dev',
    passwordHash: '$2a$10$YOUR_GENERATED_HASH_HERE',
    name: 'Developer User',
    roles: ['user'],
    permissions: ['read', 'write'],
    createdAt: new Date('2026-01-01'),
    updatedAt: new Date('2026-01-01'),
  },
];
```

### Step 3 — Restart the server

The `StaticUserRepository` is instantiated at startup — no migration required.

### Step 4 — Verify

```bash
curl -s -X POST http://localhost:3001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"developer@mono.dev","password":"mypassword"}' \
  | jq .data.user
```

## 10. Switching from Static to Drizzle Users

When you are ready to use a real database for user storage:

### Step 1 — Confirm the schema exists

`apps/api-gateway/src/database/schema/users.schema.ts` is already in place. Run migrations if not already done:

```bash
npm run db:migrate
npm run db:seed  # seeds the same static users into the DB
```

### Step 2 — Swap the repository in `auth.module.ts`

```typescript
// Before:
import { StaticUserRepository } from './repositories/static-user.repository';

// After:
import { DrizzleUserRepository } from './repositories/drizzle-user.repository';
import { DatabaseModule } from '../../database/database.module';

@Module({
  imports: [DatabaseModule],  // ← add this
  controllers: [AuthController],
  providers: [
    AuthService,
    { provide: USER_REPOSITORY, useClass: DrizzleUserRepository },  // ← swap this
  ],
  exports: [AuthService],
})
export class AuthModule {}
```

### Step 3 — Set environment variable

```bash
DATABASE_URL=postgres://user:password@localhost:5432/mono_dev
```

### Step 4 — Restart the server

Login now queries PostgreSQL instead of the in-memory array. No other files change.

## 12. Common Auth Errors

### 401 "Missing token"

The `Authorization` header is absent. Ensure all fetch calls include:
```typescript
headers: { Authorization: `Bearer ${token}` }
```

### 401 "Invalid or expired token"

The token was valid but is no longer in the session map. This happens if:
- The API server restarted (session map is cleared in-memory)
- The token was explicitly invalidated by a `POST /api/auth/logout`
- The token was rotated by `POST /api/auth/refresh` (old token is gone)

Fix: re-login to get a fresh token.

### 400 on login — "email must be an email"

The `email` field did not pass the `@IsEmail()` validator. Ensure the email is a valid format.

### 400 on login — "password must be longer than or equal to 6 characters"

The `password` field is shorter than 6 characters. The minimum length is enforced by `@MinLength(6)` on `LoginDto`.

### 401 on login — "Invalid email or password"

Either the email does not exist in `STATIC_USERS`, or the password does not match the bcrypt hash. Check the credentials and regenerate the hash if needed.
